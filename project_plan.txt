CodeSage - Production-Grade Project Plan

Version: 1.1
Date: 2026-04-23
Target build window: 15 days (MVP+)

==================================================
1. Product Goal
==================================================

CodeSage is a full-stack AI platform that analyzes source repositories and returns actionable code insights with streaming output. The system must support authenticated users, asynchronous processing, robust job tracking, and reliable result persistence.

Primary objective:
Deliver a portfolio/interview-ready distributed system that demonstrates backend architecture, AI engineering, queue design, and deployable DevOps practices.

Non-goals for this phase:
- Billing, organizations/teams, marketplace integrations
- Full enterprise multi-tenancy
- Multi-cloud production hardening

==================================================
2. System Scope and Core Capabilities
==================================================

Mandatory capabilities:
- JWT authentication (register, login, protected APIs)
- Repository ingestion for public and private GitHub URLs
- Asynchronous analysis using BullMQ + Redis
- Worker-based execution and retry management
- AI microservice (FastAPI) with RAG pipeline
- Streaming AI analysis back to frontend
- Job lifecycle tracking: pending, running, completed, failed
- Persistent storage for users, repositories, jobs, and analysis output
- Containerized deployment with docker-compose
- Logs, health checks, and baseline observability

Recommended additions (within timeline):
- Refresh token flow and token rotation
- Idempotency key for analyze requests
- Request correlation IDs for tracing across services
- Circuit breaker and timeout policy for AI service calls
- Basic metrics endpoint for queue depth and processing latency

==================================================
3. High-Level Architecture
==================================================

Flow:

Frontend (React + Vite)
  -> Backend API (Node.js + Express)
  -> BullMQ Queue (Redis)
  -> Worker Service (Node.js)
  -> AI Service (FastAPI)
  -> RAG Pipeline
  -> LLM streaming response
  -> Worker updates Backend persistence
  -> Backend streams incremental/final output to Frontend

Data stores:
- MongoDB: users, repositories, jobs, analysis results
- Redis: job queue, worker coordination, transient status cache
- ChromaDB: embeddings + vector index for retrieval

==================================================
4. Service Responsibilities
==================================================

4.1 Frontend (React + Vite)
- Auth pages and JWT session handling
- Repository submission UI (URL + optional token reference)
- Job list and status polling/streaming UI
- Live analysis viewer (SSE stream rendering)
- Error states, retries, and loading UX

4.2 Backend API (Node.js + Express)
- Auth endpoints and JWT verification
- Repository CRUD and ownership checks
- Analyze trigger endpoint to enqueue jobs
- Job read APIs (status, events, final output)
- SSE endpoint for analysis stream delivery
- Persist final outputs and metadata in MongoDB

4.3 Queue Layer (BullMQ + Redis)
- Dedicated analysis queue
- Retry/backoff policy
- Dead letter handling for exhausted retries
- Queue events for monitoring (active, completed, failed)

4.4 Worker Service (Node.js)
- Consume analysis jobs
- Update job status transitions
- Call AI service with trace metadata
- Stream chunks from AI service and forward to backend stream channel
- Store final artifacts and error details

4.5 AI Service (FastAPI)
- Accept analysis request payload (repo info + prompt/task)
- Run RAG orchestration
- Retrieve context and call LLM with streaming mode
- Return incremental tokens/chunks and final summary payload

==================================================
5. RAG Pipeline Design (Practical, Non-Naive)
==================================================

Pipeline stages:
1. Repository acquisition
- Clone repo to isolated temp workspace
- Support private repos via token-auth clone URL
- Enforce repo size/file count limits to protect resources

2. Parsing and indexing
- Parse files by language (Tree-sitter where available)
- Extract function/class/module-level segments
- Preserve file path, symbol name, language, line span metadata

3. Chunking strategy
- Prefer semantic chunks (function/class blocks) over blind fixed-size
- Fallback to adaptive token windows for unsupported languages
- Keep overlap only where needed for continuity

4. Embedding and storage
- Generate embeddings using sentence-transformers
- Store vectors and metadata in ChromaDB
- Maintain repository namespace per job/repo to prevent collision

5. Hybrid retrieval
- Dense retrieval from vector store
- Sparse retrieval via BM25 keyword index
- Merge and rerank by weighted score

6. Context expansion (optional but recommended)
- Use lightweight dependency/call graph hops
- Expand only top-k anchors to control token budget

7. Prompt assembly and LLM streaming
- Build prompt with repo metadata, top contexts, and analysis objective
- Stream output tokens/chunks to consumer as they are produced
- Capture final answer, confidence notes, and citations (file paths)

8. Execution Flow (End-to-End)
==================================================

1. User authenticates and submits repository.
2. Backend validates ownership/access and stores repo record.
3. User starts analysis via jobs endpoint.
4. Backend creates Job= pending, enqueues queue message, returns jobId.
5. Worker picks job, marks running, calls AI service.
6. AI service runs RAG pipeline and streams partial output.
7. Worker forwards stream events (Redis pub/sub or direct backend callback).
8. Backend SSE endpoint pushes updates to frontend viewer.
9. On completion, worker persists final result and marks completed.
10. On failure, worker records structured error and queue retry policy applies.

==================================================
9. Reliability and Fault Tolerance
==================================================

Queue settings:
- attempts: 3
- backoff: exponential (for example 10s, 30s, 90s)
- removeOnComplete: configurable retention
- removeOnFail: keep enough history for debugging

Timeouts:
- Backend request timeout for AI calls (worker side)
- Clone timeout and max repo size constraints
- LLM stream timeout with partial-result fallback

Failure handling:
- Distinguish transient vs terminal failures
- Move terminal failures to failed state with actionable error codes
- Preserve partial outputs when available

Idempotency:
- Support idempotency key on analyze endpoint to prevent duplicate jobs

==================================================
10. Security Baseline
==================================================

- JWT auth with short-lived access token and refresh token rotation
- Password hashing with bcrypt
- Input validation on every external endpoint (Joi/Zod)
- Encrypt sensitive secrets at rest (credentialRef)
- Never log access tokens or repository credentials
- Strict CORS configuration per environment
- Rate limit auth and analyze endpoints
- Basic secret scanning guard for logs and stored outputs

==================================================
11. Observability and Ops
==================================================

Logging:
- Structured JSON logs for all services
- Required fields: timestamp, service, level, correlationId, jobId

Health checks:
- /health for liveness
- /ready checks dependencies (Mongo, Redis, AI service)

Metrics (recommended):
- queue depth
- job throughput
- failure rate
- average end-to-end duration

Tracing (lightweight):
- Propagate correlationId from API -> queue -> worker -> AI service

17. Risks and Mitigations
==================================================

Risk: Large repo processing latency
Mitigation: file filters, size limits, incremental indexing, timeout guards

Risk: Token/security leakage from private repo handling
Mitigation: encrypted credential references, redacted logs, secret handling policy

Risk: Inconsistent streaming behavior across services
Mitigation: standardized event schema, heartbeat events, reconnect strategy

Risk: RAG quality instability
Mitigation: hybrid retrieval + reranking + evaluation samples on known repos
